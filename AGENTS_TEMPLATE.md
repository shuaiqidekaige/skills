## 代码风格

### 基本原则

- 代码实现必须结构化清晰，按职责拆分，避免将数据处理、视图渲染、事件逻辑、状态管理混在同一块代码中。
- 优先复用现有能力，避免重复实现已有的函数、组件、状态逻辑、请求逻辑和样式能力。
- 避免冗余代码、过深嵌套、过长文件和无意义封装。
- 修改代码时优先做最小必要改动，不顺手重写无关逻辑。

### 函数与组件约束

- 单个函数实现不得超过 300 行；超过时必须拆分为更小的函数或抽离到 `utils`、`hooks`、`service` 等位置。
- 单个组件实现不得超过 300 行；超过时必须拆分为子组件、hooks、布局组件或视图片段。
- 函数应尽量保持单一职责；一个函数只解决一类问题，不混合数据转换、状态修改、请求处理和 UI 控制。

### 复用与抽象规则

- 遇到相同或高度相似的实现时，优先检查项目中是否已有对应能力。
- 只有在确认现有实现无法满足需求时，才允许新增新的函数、组件或状态逻辑。
- 重复逻辑应抽象为可复用能力，避免在多个页面或模块中复制相同实现。
- 不要为了“未来可能复用”而过度抽象；出现明确复用需求后再上提到公共层。

### 命名与可读性

- 命名必须表达真实业务含义，禁止使用 `temp`、`data1`、`handleAll`、`list2` 等低语义命名。
- 文件名、函数名、组件名、hooks 名称应与实际职责一致。
- 布尔值命名应体现语义状态，如 `isLoading`、`hasPermission`、`canSubmit`。
- 目录、文件和符号命名风格应在项目内保持一致。

### 类型

- 类型定义必须完整，避免大面积使用 `any`、隐式类型逃逸或弱类型透传。

### UI 与样式实现

- UI 代码应保持结构清晰，避免 JSX 或模板中堆叠过多判断与内联处理。
- 复杂展示块应拆分为独立子组件，避免单文件堆砌全部视图逻辑。
- 样式实现应优先复用设计系统、主题变量或已有样式约定，避免重复造样式。
- 禁止在组件中散落大量魔法值；可抽离为常量、配置项或设计 token。

### 注释与提交质量

- 注释只解释“为什么这样做”，不解释显而易见的“代码正在做什么”。
- 删除或重构代码时，应同步清理无效注释、废弃逻辑和未使用代码。

## 功能模块组织

### 总体原则

- 本章节只约束目录组织和模块边界，不重复约束代码写法。
- 先判断当前能力是否属于私有功能，再决定是否需要抽到公共层。
- 某页面、某业务模块独占的逻辑，应优先放在该模块私有目录下管理。
- 只有当能力被两个及以上独立模块明确复用时，才允许上提到公共目录。
- 模块入口文件只负责组装和导出，不承担大段实现细节。

### 目录结构

推荐功能模块目录结构如下：

```text
feature-name/
  utils/
  components/
  hooks/
  layouts/
  index.tsx
```

各目录职责如下：

- `utils/`：当前模块私有的纯函数、数据转换、格式化逻辑、常量处理。
- `components/`：当前模块私有组件，仅服务于当前页面或业务模块。
- `hooks/`：当前模块私有 hooks，用于沉淀当前模块的状态逻辑、请求逻辑、交互逻辑。
- `layouts/`：当前模块私有布局、区块骨架、容器结构。
- `index.tsx`：当前模块入口文件，只做页面组装和模块装配，不堆积大段实现细节。

### 拆分规则

- 私有功能优先内聚在模块内部，不要一开始就放进全局公共目录。
- 公共目录只承载稳定、可复用、边界清晰的基础能力。
- 若某模块已经存在 `utils`、`components`、`hooks`、`layouts`，则新增内容应优先补入现有目录，而不是另起同职责目录。
- 新建目录前，先确认项目中是否已有同职责目录结构，保持风格统一。

### 反例约束

- 不要把所有逻辑都塞进 `index.tsx`。
- 不要把仅当前模块使用的组件放到全局 `components`。
- 不要把仅当前模块使用的 hooks 放到全局 `hooks`。
- 不要创建语义不清晰的目录，如 `common2`、`temp`、`newModule`。
